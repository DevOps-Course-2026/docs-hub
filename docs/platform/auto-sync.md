# 🔄 Distributed Docs & Auto-Sync

## How This Portal Works

This hub uses a **distributed documentation model**. The Docusaurus engine lives here in `docs-hub`, but the actual content is authored in the repositories where the work happens.

```text
GitHub Organization: DevOps-Course-2026
│
├── docs-hub/              ← Docusaurus engine (this repo)
│   └── docs/              ← Aggregated docs land here
│
├── course-labs-monorepo/
│   └── docs/              ← Satellite: auto-synced on push
│
├── devops-template/
│   └── docs/              ← Satellite: auto-synced on push
│
├── .github/
│   └── CONTRIBUTING.md    ← Satellite: root .md files synced on push
│
└── <any-future-repo>/
    └── docs/              ← Satellite: auto-synced on push
```

## Sync Mechanism

Each satellite repository contains a GitHub Actions workflow that triggers on push to `main` (when its `/docs` folder changes). It copies the docs into this repo under a matching subdirectory and opens or updates a PR (or pushes directly, depending on configuration).

This means:

- Docs are **authored close to the code** — in the same repo as the implementation
- This portal **never needs to be manually updated** when new content is added
- The sidebar **auto-generates** from the folder structure via `autogenerated` mode in `sidebars.ts`

## Adding a New Satellite Repo

1. Create a `/docs` folder in the satellite repo
2. Add the sync workflow below as `.github/workflows/sync-docs.yml`
3. Done — new docs appear in the sidebar automatically on next push

## Sync Workflow Template

Add this to every satellite repo. Replace `<REPO_NAME>` with the repo's name (e.g. `labs-monorepo`).

```yaml
name: Sync Docs to docs-hub

on:
  push:
    branches:
      - main
    paths:
      - 'docs/**'

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout source repo
        uses: actions/checkout@v4
        with:
          path: source

      - name: Checkout docs-hub
        uses: actions/checkout@v4
        with:
          repository: DevOps-Course-2026/docs-hub
          token: ${{ secrets.DOCS_HUB_TOKEN }}
          path: docs-hub

      - name: Copy docs
        run: |
          mkdir -p docs-hub/docs/<REPO_NAME>
          rsync -av --delete source/docs/ docs-hub/docs/<REPO_NAME>/

      - name: Commit and push
        working-directory: docs-hub
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add docs/<REPO_NAME>/
          git diff --staged --quiet || git commit -m "docs: sync from <REPO_NAME>"
          git push
```

> **Authentication:** Each satellite repo needs a secret named `DOCS_HUB_TOKEN` to push to this repo.
>
> **For this course setup:** Use a [fine-grained PAT](https://github.com/settings/tokens?type=beta) scoped only to the `docs-hub` repo with `Contents: Read & Write` permission.
>
> **At org scale:** Replace the PAT with a [GitHub App](https://docs.github.com/en/apps/creating-github-apps/about-creating-github-apps/about-creating-github-apps). A GitHub App is the correct tool here because the problem is *GitHub-to-GitHub authentication* — it has granular repo-level permissions, better audit logs, and tokens that rotate automatically. This is a different concern from a secrets manager like HashiCorp Vault, which solves *application runtime secrets* (database passwords, API keys injected into running workloads). Vault would be the right choice if, for example, a deployed service needed to fetch credentials at runtime inside Kubernetes. For a GitHub workflow pushing to another GitHub repo, a GitHub App is the right primitive.

## What Requires Manual Updates

| Item | Manual? | Reason |
| --- | --- | --- |
| Sidebar | ❌ Never | Auto-generated from folder structure |
| Blog / Lab Notes | ❌ Never | Drop `.md` file in `blog/`, push |
| `intro.md` | ❌ Never | Describes the platform, not the content |
| `docusaurus.config.ts` | ✅ Rarely | Only if tooling/org changes |
